## Project Overview

Full-stack TypeScript SaaS for supplier payment and invoice management. Stack: React 19 + Vite (frontend), Express 5 + Drizzle ORM + PostgreSQL (backend), Docker Compose for deployment.

## Commands

```bash
# Development
npm run dev            # Start Express server (development mode, serves client via Vite middleware)
npm run dev:client     # Start Vite dev server only (port 5000)

# Production build
npm run build          # Build client (Vite → dist/public/) + server (esbuild → dist/index.cjs)
npm start              # Run production build

# Database
npm run db:push        # Apply schema changes via drizzle-kit

# Type checking
npm run check          # TypeScript type check (no emit)
```

```bash
# Docker (development)
docker compose -f docker-compose.dev.yml up --build

# Docker (production)
docker compose -f docker-compose.prd.yml up --build -d
```

There is no test suite and no linter configured.

## Architecture

### Request Flow

1. All traffic enters Express (`server/index.ts`)
2. In development, `server/vite.ts` serves the React app via Vite middleware (HMR included)
3. In production, `server/static.ts` serves pre-built files from `dist/public/`
4. API routes are defined in `server/routes.ts` and protected by `isAuthenticated` middleware from `server/auth.ts`

### Key Server Files

- **`server/auth.ts`** — Passport.js Local Strategy with bcryptjs. Sessions stored in PostgreSQL via `connect-pg-simple`. 7-day TTL. Exposes `/api/register`, `/api/login`, `/api/logout`, `/api/auth/user`.
- **`server/routes.ts`** — All business logic endpoints: suppliers, payments, file uploads (multer, 10MB limit), email (Resend API), Excel export (XLSX).
- **`server/storage.ts`** — Database access layer implementing `IStorage` interface. Handles encryption/decryption of sensitive fields (`resendApiKey`, `gmailRefreshToken`) using `server/crypto.ts`.
- **`server/crypto.ts`** — AES-256-GCM encryption. Key derived from `ENCRYPTION_KEY` env var. Format: `{iv}:{tag}:{encrypted}` (hex).
- **`server/db.ts`** — Drizzle ORM connection via `node-postgres` pool using `DATABASE_URL`.

### Shared Schema (`shared/schema.ts`)

Single source of truth for database tables (Drizzle) and validation (Zod via drizzle-zod). Both server and client import from here. Key tables: `users`, `suppliers`, `payments`, `services`, `files`, `sessions`.

### Frontend Structure

- **Router**: Wouter (not React Router)
- **Server state**: TanStack Query (React Query) — all API calls go through `client/src/lib/queryClient.ts`
- **Auth state**: `client/src/hooks/use-auth.ts` — wraps `/api/auth/user` query and login/register/logout mutations
- **UI**: Radix UI primitives + Tailwind CSS 4 + Shadcn-style components in `client/src/components/ui/`
- **Forms**: React Hook Form + Zod (schemas re-used from `shared/schema.ts`)
- Path alias `@` maps to `client/src/`, `@shared` maps to `shared/`

### Environment Variables

Required variables (see `.env.dev` for dev values):

| Variable | Purpose |
|---|---|
| `DATABASE_URL` | PostgreSQL connection string |
| `SESSION_SECRET` | Express session signing |
| `ENCRYPTION_KEY` | AES-256-GCM key (min 32 chars) |
| `PORT` | Server port (default 3000) |
| `NODE_ENV` | `development` or `production` |

### Build Output

- `dist/public/` — Vite-built React client (static files)
- `dist/index.cjs` — esbuild-bundled Express server (CommonJS)

### Database Migrations

Migrations are generated by `drizzle-kit` and stored in `migrations/`. To make schema changes: edit `shared/schema.ts`, then run `npm run db:push`.

## Regras Operacionais

Regras obrigatórias que DEVEM ser seguidas em todas as operações:

1. **SEMPRE usar `docker-compose.dev.yml` para rebuild e deploy** — NUNCA usar `docker-compose.prd.yml` salvo instrução explícita do Ricardo.

2. **Antes de qualquer rebuild, confirmar qual compose file será usado** — não iniciar o build sem confirmação explícita do arquivo.

3. **NUNCA declarar sucesso sem verificar o status dos containers** — após qualquer rebuild, executar obrigatoriamente:
   ```bash
   docker compose -f docker-compose.dev.yml ps
   ```
   e confirmar STATUS `Up` para ambos os containers antes de declarar sucesso.

4. **Após rebuild, sempre verificar os logs da aplicação** — executar obrigatoriamente:
   ```bash
   docker compose -f docker-compose.dev.yml logs app --tail=20
   ```
   e confirmar que o app subiu sem erros.

5. **Git push sempre para branch `main`** — nunca fazer push para outra branch salvo instrução explícita.

6. **Prompts de correção devem ser explícitos e sem margem para interpretação** — nunca assumir qual arquivo ou ambiente usar; sempre especificar explicitamente o arquivo, o ambiente e o compose file envolvidos.
